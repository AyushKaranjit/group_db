(3:30 - 6:00) Constraints & Integrity (Ayush)

(Ayush): "Thank you, Nabin. Once the tables were defined, my focus was on implementing constraints to ensure database integrity, which is crucial for maintaining data accuracy and consistency. We focused on three main types: Entity, Referential, and Domain Integrity."

(Ayush): (Show/Scroll through constraints_group1.txt from ayush.html) "This constraints_group1.txt script uses ALTER TABLE statements to add these constraints after the tables are created."

(Ayush): "For Entity Integrity, we defined Primary Keys for each table, like pk_students on students(student_id). This ensures every student has a unique ID and no record is duplicated." (Point to relevant PK constraint)

(Ayush): "For Referential Integrity, we used Foreign Keys to link related tables. For instance, fk_students_degrees links the degree_id in the students table to the degree_id in the degrees table." (Point to relevant FK constraint) "This prevents us from assigning a student to a non-existent degree or deleting a degree that still has students assigned to it."

(Ayush): "For Domain Integrity, we used several constraint types:

  • NOT NULL constraints, like on student_fname, ensure essential information isn't missing." (Point to a NOT NULL constraint)
  • CHECK constraints validate data values. For example, check_grade on student_submissions ensures grades are within the allowed set ('A+', 'A', etc.), and check_module_status ensures the status is 'Y' or 'N'." (Point to CHECK constraints)
  • UNIQUE constraints, like unique_assignment_code on assignments, prevent duplicate assignment codes." (Point to a UNIQUE constraint)
  • DEFAULT constraints provide default values, like setting the attendance_date in the attendances table to SYSDATE if not specified." (Point to a DEFAULT constraint)

(Ayush): "These constraints work together to keep the database reliable. Next, Utsab will show how we populated these tables with data."

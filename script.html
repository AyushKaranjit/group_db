(0:45 - 3:15) Database Design & Table Creation (Nabin)

(Nabin): "Thanks, Ayush. Based on the ERD, I developed the creates_group1.txt script. This defines all our tables like students, modules, degrees, etc., specifying the columns and their datatypes (NUMBER, VARCHAR2, DATE) according to the table specifications we designed."

(Nabin): "Equally important is the drops_group1.txt script. This script is crucial for resetting the database cleanly. It's structured very deliberately:"

  • "First, it drops all Foreign Key constraints. We do this first because these constraints create dependencies between tables. Trying to drop a table that's referenced by a foreign key in another table would cause an error."
  • "Next, it drops the Primary Key and Unique constraints. These need to go before the tables themselves."
  • "Then, it drops the actual tables. Now that the constraints linking them are gone, they can be removed without issues."
  • "Finally, it uses PURGE RECYCLEBIN;. This permanently removes the dropped tables from Oracle's recycle bin, ensuring a completely clean state, which is good practice, especially during development."
"This specific order prevents dependency errors and ensures the database can be reliably reset. Now, Ayush will explain how we added the integrity rules using constraints."

(3:15 - 5:45) Constraints & Integrity (Ayush)

(Ayush): "Thanks, Nabin. My task was to implement the constraints defined in our table specifications using the constraints_group1.txt script. These rules are vital for maintaining data accuracy and consistency – ensuring Entity, Referential, and Domain integrity."

(Ayush): "We used ALTER TABLE commands to add these rules after the tables were created.

  • Primary Keys (like pk_students) ensure every record in a table is unique – that's Entity Integrity.
  • Foreign Keys (like fk_students_degrees) link related tables and ensure relationships are valid, preventing 'orphan' records – that's Referential Integrity. For example, a student must be linked to an existing degree.
  • For Domain Integrity – ensuring data values are valid – we used:
    • NOT NULL to make sure essential data isn't skipped.
    • CHECK constraints to limit values to acceptable ranges or sets (like grades 'A+', 'A', etc., or module status 'Y'/'N').
    • UNIQUE constraints to prevent duplicates in columns like assignment codes.
    • DEFAULT constraints to provide automatic values, like setting attendance dates to the system date (SYSDATE).
"These constraints work together to enforce the business rules and keep the database reliable. Next, Utsab will show how we populated these tables."

(5:45 - 7:15) Data Insertion (Utsab)

(Utsab): "Thanks, Ayush. With the structure and rules ready, I used the inserts_group1.txt script to add sample data. This script uses standard INSERT INTO statements."

This data is essential for testing that our constraints work correctly – for example, ensuring we can't insert a student submission for a non-existent student or submission."

(7:15 - 9:30) Querying the Database (Anuskar)

(Anuskar): "Alright, thanks Utsab. To retrieve and analyze the data, I developed the queries_group1.txt script. This demonstrates how to extract useful information."

(Anuskar): "The script includes various query types:

  • Simple SELECT * queries to view all data in a table.
  • Projection queries using SELECT with specific column names, often combined with ORDER BY for sorting.
  • Restriction queries using the WHERE clause with different conditions like equality (=), patterns (LIKE), negation (!=), logical operators (AND, OR), and date ranges (BETWEEN).
  • Join queries (2-table and 3-table examples) using JOIN...ON to combine data from related tables, allowing us to see, for instance, student names alongside their degree titles or module details with examiner names.
  • We also included examples using aggregate functions like COUNT with GROUP BY to summarize data, such as counting submissions per module.
"These queries show the flexibility of SQL in retrieving specific, combined, or summarized information from the database."

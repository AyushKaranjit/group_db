Creates Script (About 1 minute)

Nabin: Thank you, Utsab. I worked on the scripts that create and remove the database tables.

Nabin: Now Ayush will explain how we made sure the data stays correct with constraints.

Constraints Script (About 2 minutes)

Ayush: Thank you, Nabin. After setting up the tables, we needed to add rules to make sure the data stays correct. We did this with the constraints_group1.sql script.

(shows script) This script adds rules that the database will follow at all times.

We added several types of rules:

  • Primary Keys for each table to make sure every record can be found easily. For example, each student has a unique student_id.
  • Foreign Keys to connect tables together. These make sure that when a student is linked to a degree program, that degree must exist in the degrees table.
  • Check constraints to limit what values can go in certain columns. For instance, the grade column can only have proper letter grades.
  • Not Null constraints for important information that must be filled in. Things like a student's first name cannot be left empty.
  • Unique constraints for information that shouldn't be repeated, like assignment codes.
  • Default values for when information isn't provided. For example, attendance date will use today's date if none is given.

Ayush: These rules work together to keep our data accurate and reliable. Now Utsab will explain how we added sample data to our tables.

Insert Script (About 1.5 minutes)

Utsab: Thank you, Ayush. After setting up the tables and rules, we needed to add sample data to the database using the inserts_group1.sql script.

(shows script) This script contains INSERT INTO statements for each table in our database. The assignment asked for at least five records per table, but we added extra in some tables to make our query examples better.

Utsab: We used realistic test data with proper names, addresses, grades, and course codes. Using realistic data helps us test how the database would work in a real school setting.

Utsab: We made sure to format dates correctly as the assignment required. We used the format DATE '2023-09-15' for all date information.

Utsab: We also used a different way to add data i.e to specify just certain columns, like the ID and name for a marker. This lets the database handle the other columns, using defaults or leaving them empty.

Utsab: We also demonstrated inserting data while relying on default constraints. For example, when adding an attendance record, we did not specify the attendance date. Because we set up a default value for that column, the database automatically used the current system date (SYSDATE) for the record.

Utsab: The main challenge was keeping all relationships between tables correct. When adding a student enrollment, we had to make sure both the student and the module existed in their tables first.

Utsab: I'll now hand over to Anuskar, who will show how we get useful information from our database.

Queries Script (About 2 minutes)

Anuskar: Thank you, Utsab. Now I'll show how we can get useful information from our database using the queries_group1.sql script.

(shows script) This script shows different ways to get information using SQL. Here are the main types of queries we made:

  • Simple Select queries that get all data from one table. For example, SELECT * FROM students; shows all student records.
  • Projection queries that get only certain columns. This is useful when you only need specific information, like student names and phone numbers.
  • Filter queries that use the WHERE clause to find specific records. For example, finding students who live in a certain city or assignments due this month.
  • Join queries that combine information from multiple tables. These show connections like which students are taking which modules, or which modules are part of each degree program.
  • Advanced queries using functions like COUNT, AVG, and grouping with GROUP BY. These help answer questions like how many students are in each module or what the average grade is.

Anuskar: These queries show how our database can answer practical questions that school staff might need to know.

Anuskar: Now Nabin will explain the cleanup script we created.

Drops Script (About 30 seconds)

Nabin: I also made a cleanup script called drops_group1.sql. (shows script) This script helps us restart the database when needed during testing. It removes all constraints first, then removes all tables, and finally cleans up the recyclebin. This gives us a clean start each time we run our scripts.

Nabin: The order is important - we must remove the constraints before the tables to avoid errors. This keeps everything working smoothly when we rebuild the database.

Nabin: Finally, Utsab will explain how we put everything together with the master run script.

Master Run Script (About 1 minute)

Utsab: To make testing our database easy, we created a main script called run_group1.sql.

(shows script) This script runs all our other scripts in the right order. It first cleans everything with the drops script, then creates the tables, adds the rules, inserts the data, and finally runs the queries.

Utsab: This approach makes it simple to test the whole database with just one command. You don't need to know which script to run first - the master script handles everything in the correct order.

Utsab: It also helped us during development. When we changed one script, we could run the master script to make sure everything still worked together properly.
